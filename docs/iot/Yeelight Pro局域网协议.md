# 一、系统概述

Yeelight Pro照明控制系统由硬件、云端、用户App组成。硬件包括网关、灯具、传感器、面板、全面屏等组成。网关通过蓝牙MESH协议与其他蓝牙mesh子设备进行通信，通过Wi\-Fi或者有线与云端/本地服务器通信。App通过局域网或者服务器与网关通信，并实现设备的控制。

- mesh网关：⼀个特殊硬件设备，所有系统的其他mesh设备统称为mesh⼦设备，⽽与之相对的就是mesh⽹关， 称为桥设备。⼦设备本⾝只有蓝⽛通信能⼒，⽽桥设备同时具备蓝⽛和WiFi和以太⽹能⼒。桥设备是连接互联⽹和蓝⽛MESH⽹络的桥梁，在中间承担协议翻译，信息存储和转发的功能。完成入网操作后，网关一方面注册到Yeelight Pro云端系统中并与用户账户进行绑定；另一方面会接入到mesh子设备所在的mesh网络，进行设备和云端之间的协议翻译/转发，并可以本地执行定时/设备联动等。

- mesh子设备：mesh子设备本身只有蓝牙通信能力，入网后可接入mesh网络。

- 全面屏：全面屏是一款带屏幕的Wi\-Fi设备，配备两个继电器，完成入网操作后，设备会注册到Yeelight Pro云端系统中并与用户账户进行绑定，全面屏与mesh网关通过本地局域网通信进行设备情景操作。

Yeelight Pro提供本地控制协议，第三方设备可以通过局域网发现协议扫描发现附近的mesh网关，通过建立TCP连接，对mesh子设备或全面屏设备进行操作。整个流程概括如下：

本地发现网关\-\-\>建立socket连接\-\-\>获取网关设备拓扑关系\-\-\>设备控制/状态同步等。

# 二、网关发现与连接

第三方设备连接网关所在的网络，通过发送UDP广播发现网关，然后通过控制TCP连接与网关建立会话。

***注意：网关需要开启局域网******控制******模式才能被发现，该功能在Yeelight Pro应用******，******网关******设置******界面开启，新入网设备******局域网控制******功能默认******关闭******。***

## UDP广播格式

广播地址：网络中广播地址

目标端口：1982

消息文本：YEELIGHT\_GATEWAY\_CONTROL\_DISCOVER

## Response数据解析

网关收到广播后，立即发送回应给消息源，格式如下：

```Apache
pid:1
mac:F8:24:41:00:23:A4
did:22535
ip:192.168.1.101
```

其中

|字段|含义|
|---|---|
|pid|产品id，目前支持两款产品网关（1）和全面屏（2）|
|mac|设备mac地址|
|did|设备uuid，全局唯一|
|ip|设备ip地址|

## 建立TCP连接

第三方设备收到回应后，可解析到网关IP地址，通过TCP建立socket网络连接，网关端口号为65443。



*注意：可以使用telnet命令手动发送/接收网关消息，以测试连通性*

```Apache
*telnet <ip address> 65443*
```



# 三、网关控制协议

Yeelight Pro 局域网控制API用于从本地网关通过TCP连接获取到蓝牙mesh网络内的设备信息，同时提供设备控制的能力。三方设备和网关建立TCP连接后，可发送JSON格式的指令进行设备枚举/设备控制；当设备状态发生变化或产生事件时，网关将主动推送JSON格式的消息给第三方应用。

## 消息内容

### 1\.1 消息格式

客户端和网关之间使用JSON格式的消息进行交互，消息基于TCP协议进行传递；消息之间以“\\r\\n”作分隔。

**1\.1\.1 ****请求消息的基本格式如下：**

```JSON
{ 
  "version": "1.0", 
  "id": 123, 
  "method": "gateway_set.prop"
} 
```

字段定义：

|字段|含义|
|---|---|
|version|协议版本，可选（默认1\.0），目前只有一个可用版本1\.0|
|id|请求ID，请求消息与回应消息的id相同，可根据这个来匹配|
|method|方法名|

**1\.1\.2 ****部分请求消息可以收到回应消息，基本格式如下：**

```Prolog
{ 
  "id": "uuid",
  "result": "ok",
  "data": {
    ...
  }
} 
```

字段定义：

|字段|类型|含义|
|---|---|---|
|id|int32|请求ID，请求消息与回应消息的id相同，可根据这个来匹配|
|result|string|执行结果<br>ok: 代表成功<br>error: 代表失败|
|data|json|附加信息|

**1\.2 ****目前****已****支持的上下行消息的方法有如下几种：**

|方向|method||comment|**是否返回1\.1\.2节所述ACK**|
|---|---|---|---|---|
|网关\-\>客户端<br>|gateway\_post\.topology||通报网关下面可以独立控制的子设备或设备组|NA|
||gateway\_post\.prop||通报子设备或设备组的状态|NA|
||gateway\_post\.event||通报传感器触发的事件|NA|
|客户端\-\>网关<br><br>|gateway\_get\.topology||请求网关下面可以独立控制的子设备或设备组|N|
||gateway\_set\.prop||设备控制（子设备、设备组、情景等）|Y|
||gateway\_get\.node||请求单个设备信息|N|
||gateway\_get\.group||请求Mesh灯组信息|N|
||gateway\_get\.room||请求房间信息|N|
||gateway\_get\.scene||请求情景信息|N|
||gateway\_set\.event||模拟虚拟传感器设备事件触发|Y|

接下来的章节详细介绍每种方法的格式及使用方法。

### 1\.2 网关\-\>客户端

#### 拓扑同步消息

1. 网关触发

    当mesh网络中设备或组信息发生变化时（如增删设备、改名、分组建立与解除等），网关主动向所有已连接客户端发送最新的拓扑同步消息及状态同步信息

2. 客户端触发

    当客户端连上网关后，请求拓扑同步；网关收到后立即回应拓扑同步消息



拓扑同步消息包含网关下可见/可控制的最小单元（设备分组，未被分组的设备），代码示例

```JSON
    "roomid":345 //普通版需网关升级到3.4.2.001，增强版网关需升级到5.1.25.001
```

#### 状态同步消息

状态同步消息用来向客户端同步mesh网络下设备和分组的状态。状态同步消息可能包含所有的节点的全部状态或者发生变化的状态

1. 当客户端初次连上网关并发起拓扑同步请求后，网关在发送拓扑同步回应后立即发送节点状态的全量同步

2. 网关每隔十分钟向所有客户端发起全量同步

3. 当节点状态发生变化后，网关发送状态变化同步消息 （频率不高于两秒一次，多个状态变化可能合并）

状态同步消息，代码示例：

```JSON
{
    **"method"**:**"gateway_post.prop"**,
    **"id"**:**1**,
    **"nodes"**:[
        {
            **"id"**:**1001**,
            **"nt"**:**2**,
            **"o"**:**true**,
            **"fv"**:**"1.0.1"**,
            **"params"**:{
                **"p"**:**true**,
                **"l"**:**20**,
                **"c"**:**255**,
                **"ct"**:**4000**,
                **"m"**:**1**
            }
        }
    ],
    **"scenes"**:[
        {
            **"id"**:**413**,
            **"n"**:**"回家模式"**,
            **"params"**:{
                **"state"**:**"active" //"inactive", "unknown"**
            }
        }
    ]
}
```

其中：

|字段|取值范围|含义|
|---|---|---|
|nodes|json数组|mesh网关下所有未分组和分组的设备信息|
|id|64位的无符号数值类型的10进制表示|节点ID，对某种nt来说，节点ID全局唯一|
|nt<br>|- 房间（1）<br>- Mesh子设备（2）<br>- 区域（3）<br>- Mesh组（4）<br>- 房屋/整屋（5）<br>- 情景（6）|node type缩写，节点类型|
|o|- true<br>- false|online缩写，true代表设备在线，false代表设备离线|
|fv|x\.x\.x|firmware version缩写，设备当前固件版本|
|params|json格式文本|设备当前属性信息，详细查看关键字对照|

当控制面板发生点按事件或传感器设备被触发并通知网关，网关向所有客户端发送事件通知消息。

通知消息格式，代码示例：

```JSON
{
  "id": 13810,
  "method": "gateway_post.event",
  "nodes": [{
    "params": {},
    "value": "motion.true",
    "id": 1552,
    "nt": 2
  }]
}
```

其中：

|字段|取值范围|含义|
|---|---|---|
|nodes|json数组|mesh网关下所有未分组和分组的设备信息|
|id|64位的无符号数值类型的10进制表示|节点ID，对某种nt来说，节点ID全局唯一|
|nt|- 房间（1）<br>- Mesh子设备（2）<br>- 区域（3）<br>- Mesh组（4）<br>- 房屋/整屋（5）<br>- 情景（6）|node type缩写，节点类型<br>|
|value|- 有人移动（motion\.true）<br>- 无人移动（motion\.false）<br>- 门打开（contact\.open）<br>- 门关闭（contact\.close）<br>- 门磁被拆下（contact\.alarm）<br>- 门磁被安装上（contact\.normal ）<br>- 面板被点击（panel\.click ）<br>- 面板按键被按住（panel\.hold）<br>- 面板按键被松开（panel\.release）<br>- 旋钮旋转（knob\.spin）<br>- 有人靠近（approach\.true）<br>- 有人远离（approach\.false）<br>- 有人挥手（handwave）|事件名称，注意：<br>无人移动是指检测到有人移动后，一段时间内连续检测不到移动则触发一次|

#### 请求单个设备信息

客户端连上网关后，可以发送gateway\_get\.node命令来请求单个Mesh节点设备的信息，命令入参为Mesh设备节点id，id为0表示获取连接网关所有节点信息。

请求消息示例，获取节点id为1270的设备信息，代码示例：

```JSON
{
  "id": 123,
  "method": "gateway_get.node",
  "params": {
```

#### 模拟虚拟传感器事件输入

用于模拟传感器事件输入，如：模拟人体传感器触发有人事件、模拟情景面板触发短按事件、模拟门窗传感器打开事件等，可发送gateway\_set\.event方法模拟，代码示例：

```JSON
{
    "id":4082,
    "method":"gateway_set.event",
    "nodes":[{
        "params": {
            "key":3,
            "count":1
            },
        "value":"panel.click",
        "id":331915,
        "nt":2
    }]
}
```

其中：

|字段|取值范围|含义|
|---|---|---|
|nodes|Json 数组|mesh网关下所有未分组和分组的设备信息|
|id|64位的无符号数值类型的10进制表示|节点ID，对某种nt来说，节点ID全局唯一|
|nt|- 房间（1）<br>- Mesh子设备（2）<br>- 自定义分组（3）<br>- Mesh组（4）<br>- 房屋/整屋（5）<br>- 情景（6）|node type缩写，节点类型|
|value|- 有人移动（motion\.true）<br>- 无人移动（motion\.false）<br>- 门打开（contact\.open）<br>- 门关闭（contact\.close）<br>- 门磁被拆下（contact\.alarm）<br>- 门磁被安装上（contact\.normal ）<br>- 面板被点击（panel\.click ）<br>- 面板按键被按住（panel\.hold）<br>- 面板按键被松开（panel\.release）<br>- 旋钮旋转（knob\.spin）|事件名称，注意：<br>无人移动是指检测到有人移动后，一段时间内连续检测不到移动则触发一次|
|params|json格式文本|设备当前属性信息，详细查看关键字对照|

# 四、wifi 全面屏的局域网发现与控制协议

Yeelight全面屏开关（简称全面屏）的本地控制基本原理和pro网关相同。第三方设备可以通过局域网发现协议扫描发现附近的全面屏设备，然后设备与全面屏建立TCP连接，后续所有操作都建立在TCP通讯的基础上。阅读本节前请先阅读前面章节了解基本原理。

全面屏本地控制与网关的差别有如下几点：

1. 发现命令，广播发现文本改为：YEELIGHT\_DEVICE\_CONTROL\_DISCOVER；

2. 局域网发现协议，网关回复pid为1，全面屏回复pid为2；

3. 网关下设备拓扑是动态的，根据网关下实际mesh设备动态调整。全面屏包含固定的2个子设备：情景按键和双路控制器，且二者共用同一resid。

4. 全面屏本地控制支支持如下操作：控制双路开关，接收情景按键按下消息和双路状态上报。

#### 本地发现

广播发送成功后，可以收到相应消息如下：

```JSON
pid:2
mac:F8:24:41:00:23:A4
did:22535
ip:192.168.1.101
```

其中：

|字段|含义|
|---|---|
|pid|产品id，目前支持两款产品网关（1）和全面屏（2）|
|mac|设备mac地址|
|did|设备uuid，全局唯一|
|ip|设备ip地址|

#### 拓扑获取

获取命令：

```JSON
{
    **"id"**:**1**,
    **"method"**:**"device_get.topology"**
}
```

全面屏回复：

```JSON
{ 
    "id": 68, 
    "method": "getway_post.topology", 
    "nodes": [
        { 
            "id": 7919, 
            "n": "dual",

            "type": 7 
        }, 
        { 
            "id": 7919,

            "nt": 2, 
            "type": 128 
        } ] 
}
```

```JSON
{
    "id":2, 
    "method":"device_set.prop", 
    "nodes":[
        {
            "id":7919,
            "nt":2,
            "set":
                {
                    "1-p":false
                }
         }]
}
```

其中nodes里面参数解释如下：

|字段|取值范围|含义|
|---|---|---|
|id|64位的无符号数值类型的10进制表示|需控制的设备节点ID，可从拓扑获取返回中获得|
|nt|- 2|节点类型，可从拓扑获取返回中获得，仅支持继电器|
|set|json字段<br>|期待设置的属性和目标值 （可选）<br>其中，全名屏自带双路继电器，属性和双路继电器一致，可以参考2\.1\.3<br>1\-p：双路开关左边一路，2\-p:  双路开关右边一路，p：双路总开关|
|toggle|json字段|期待执行取反操作的属性列表，仅支持继电器（可选）|
